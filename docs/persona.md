# Persona v1

## Persona 1

- นรินทร์
- นักศึกษามหาวิทยาลัย / เจ้าของธุรกิจขนาดเล็ก


### ความเป็นมา

- นรินทร์เป็นนักศึกษามหาวิทยาลัย  / เจ้าของธุรกิจขนาดเล็กที่ดูแลเว็บไซต์อย่างน้อยหนึ่งแห่ง (เช่น ระบบสตาร์ทอัพขนาดเล็ก)
เขาไม่ใช่ผู้เชี่ยวชาญด้าน DevOps และตรวจสอบระบบเฉพาะเมื่อมีปัญหาเกิดขึ้นแล้ว เว้นแต่จะมีอะไรเสียหาย


### ระดับทางเทคนิค

- ความรู้ทางเทคนิคพื้นฐานถึงระดับกลาง
- ามารถปรับใช้บริการโดยใช้แพลตฟอร์มต่างๆ เช่น Vercel, AWS หรือ Docker
- เข้าใจ URL และแนวคิดพื้นฐานเกี่ยวกับเซิร์ฟเวอร์
- ไม่เข้าใจสแต็กการตรวจสอบ, Prometheus หรือเมตริกคลาวด์อย่างลึกซึ้ง

### เป้าหมาย

1. รู้ว่าเว็บไซต์หรือบริการของเขาล่มได้อย่างไร
2. เข้าใจสุขภาพระบบในแง่ที่
    - ง่าย
    - พื้นฐาน
3. เหมาะสมกับโครงการเล็กๆ
4. ข้อความแสดงข้อผิดพลาดที่ชัดเจนเข้าใจง่าย
5. รู้ได้อย่างชัดเจนต้องทำอย่างไรเมื่อมีอะไรผิดพลาด

### ความหงุดหงิด / Pain point

1. บริการดูเหมือนจะ "ดี" จนกระทั่งจู่ๆ ก็ล่มโดยไม่ทราบสาเหตุ
2. เครื่องมือตรวจสอบที่มีอยู่รู้สึกว่า:
    - ซับซ้อนเกินไป
    - เทคนิคเกินไป
3. เกินความจำเป็นสำหรับโครงการเล็กๆ
4. ข้อความแสดงข้อผิดพลาดและตัวชี้วัดยากต่อการตีความ
5. ไม่รู้ว่าจะทำอย่างไรเมื่อมีอะไรผิดพลาด


### ความต้องการ

1. วิธีง่ายๆ ในการเพิ่มบริการ (URL หรือ /metrics endpoint)
2. ไฟแสดงสถานะชัดเจน (Up/Down/Slow/Warn)
3. ข้อมูลสุขภาพพื้นฐานโดยไม่ต้องตั้งค่าลึก
4. คำแนะนำง่ายๆ เกี่ยวกับสิ่งที่อาจผิดพลาดและสิ่งที่ควรตรวจสอบต่อไป


### Ayser ช่วยได้อย่างไร
Ayser เป็นแพลตฟอร์มน้ำหนักเบาที่นรินทร์สามารถ:
1. ลงทะเบียนเว็บไซต์หรือบริการของเขาโดยใช้ URL
2. ดูสถานะปัจจุบันและตัวชี้วัดสุขภาพพื้นฐาน
3. เข้าใจสภาพระบบได้โดยไม่ต้องอ่าน metric
4. รักษาความน่าเชื่อถือในการให้บริการโดยไม่ต้องมีความรู้เชิงลึกด้าน DevOps


## Persona 2

- สมชาย
- อาจารย์ / ผู้ดูแลระบบเว็บไซต์องค์กร

### ความเป็นมา

- สมชายเป็นอาจารย์มหาวิทยาลัยที่ได้รับมอบหมายให้ดูแลเว็บไซต์และระบบสารสนเทศหลายระบบของคณะหรือมหาวิทยาลัย
เช่น เว็บไซต์หลัก, ระบบลงทะเบียน, ระบบภายใน, หรือบริการออนไลน์สำหรับนักศึกษา
เขาไม่ได้มีตำแหน่ง DevOps โดยตรง และการดูแลระบบเป็น “งานแถม” จากหน้าที่หลักด้านการสอนและงานวิชาการ เมื่อเกิดปัญหา เขามักเป็นคนแรกที่ถูกติดต่อ แม้จะไม่ใช่คนที่สร้างระบบทั้งหมดด้วยตนเอง


### ระดับทางเทคนิค

1. ความรู้ทางเทคนิคระดับกลางถึงค่อนข้างสูง
2. สามารถ:
    - Deploy ระบบด้วย Vercel, AWS, Azure หรือ Docker
    - เข้าใจแนวคิด server, resource, uptime
3. ไม่ได้:
    - มีเวลาตั้งค่า monitoring เชิงลึก
    - มีเวลาวิเคราะห์ metric จำนวนมากแบบ real-time

### เป้าหมาย

1. เห็นภาพรวมสุขภาพของหลายระบบได้ในที่เดียว
2. ตรวจสอบสถานะและ uptime ได้อย่างรวดเร็ว
3. อธิบายสถานการณ์ระบบให้:
    - ผู้บริหาร
    - เพื่อนอาจารย์
    - เจ้าหน้าที่ที่ไม่ใช่สายเทคนิคเข้าใจได้ง่าย

### ความหงุดหงิด / Pain point

1. ระบบมีจำนวนมาก แต่เวลามีจำกัด
2. เครื่องมือ monitoring:
    - ต้องตั้งค่าเยอะ
    - ข้อมูลเยอะเกินจำเป็น
    - ยากที่จะอธิบายปัญหาระบบให้คนที่ไม่เข้าใจเทคนิค
3. เมื่อเกิดปัญหา:
    - รู้ว่ามีปัญหาแต่ไม่รู้จะ “สรุปให้คนอื่นฟัง” อย่างไร

### ความต้องการ

1. เครื่องมือที่:
    - รวมสถานะหลายระบบไว้ในแดชบอร์ดเดียว
    - แสดงผลแบบเข้าใจง่าย (ปกติ / ช้า / มีปัญหา)
2. ข้อมูลสุขภาพระบบในระดับสรุป
3. ข้อความอธิบายที่สามารถนำไปสื่อสารต่อได้ทันที
4. ไม่ต้องตั้งค่าซับซ้อนหรือดูแลตลอดเวลา

### Ayser ช่วยได้อย่างไร
Ayser ช่วยสมชายโดย:
1. แสดงภาพรวมสุขภาพของหลายระบบในหน้าเดียว
2. สรุปสถานะระบบในรูปแบบที่เข้าใจง่าย
3. ลดภาระการอธิบายปัญหาให้คนที่ไม่ใช่สายเทคนิค
4. ช่วยให้การดูแลระบบเป็นเรื่อง “ควบคุมได้” แม้มีเวลาจำกัด
