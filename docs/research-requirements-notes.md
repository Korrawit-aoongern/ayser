# Research Requirements Notes

## In-scope research & user problems (to be validated)

### How much system health information is enough for users

จากการศึกษางานวิจัยและแนวคิดด้าน system health monitoring พบว่า
ผู้ใช้งานที่ไม่ใช่ DevOps หรือผู้ดูแลระบบเชิงลึก ไม่ต้องการข้อมูลจำนวนมาก
แต่ต้องการข้อมูล “เพียงพอ” เพื่อช่วยตัดสินใจได้อย่างรวดเร็วว่า
ระบบยังอยู่ในสถานะที่ปลอดภัยหรือควรกังวล

ระดับข้อมูลขั้นต่ำที่ผู้ใช้งานคาดหวังโดยทั่วไป ได้แก่

1. ความสามารถในการทราบล่วงหน้าว่าระบบกำลังมีแนวโน้มผิดปกติ
   เพื่อหลีกเลี่ยงเหตุขัดข้องร้ายแรง (early warning)

2. ความมั่นใจว่าระบบหรือบริการยังคงทำงานอยู่ (availability)

3. การรับรู้ภาระการทำงานของระบบในระดับสรุป
   เช่น CPU, Memory หรือ Network ในเชิงสถานะ
   โดยไม่จำเป็นต้องแสดงค่าทางเทคนิคเชิงลึก

4. การมองเห็นสถานะของบริการที่ระบบพึ่งพา
   เช่น ฐานข้อมูล หรือบริการภายนอกที่สำคัญ (dependencies)

5. การตรวจสอบว่ากระบวนการหรือการทำงานเบื้องหลังที่สำคัญ
   ยังคงทำงานตามปกติหรือไม่ (background processes)

6. การรับรู้รูปแบบการใช้งานโดยรวมของระบบ
   เพื่อช่วยบ่งชี้พฤติกรรมผิดปกติหรือแนวโน้มที่ควรปรับปรุง
   โดยไม่มุ่งเน้นการวิเคราะห์เชิงธุรกิจหรือ analytics เชิงลึก

ข้อมูลข้างต้นถือเป็นกรอบขั้นต่ำของข้อมูลสุขภาพระบบที่ผู้ใช้งานทั่วไปสามารถเข้าใจและนำไปใช้ได้จริงโดยไม่สร้างภาระด้านการตีความหรือความรู้เชิงเทคนิค

Ayser ถูกออกแบบให้เป็น  
**Application-Level Monitoring Platform**  
ที่ให้ข้อมูลในระดับที่เข้าใจง่าย  
พร้อม *optional infrastructure insights*  
เฉพาะในกรณีที่ผู้ใช้งานสามารถและต้องการให้ข้อมูลเพิ่มเติม

## Out of Scope (Current Phase)

ในเฟสปัจจุบัน Ayser **ยังไม่ครอบคลุม** ขอบเขตต่อไปนี้ เพื่อรักษาความเรียบง่ายของระบบ
และโฟกัสไปที่การประเมินสุขภาพระบบในระดับพื้นฐาน

- ระบบแจ้งเตือน (Alert) แบบ real-time  
  > ปัจจุบันรองรับเพียงการแจ้งเตือนพื้นฐานที่ไม่ใช่ real-time

- การเชื่อมต่อ Cloud Provider ผ่าน OAuth  
  รวมถึงการรองรับ URL และโครงสร้าง API แบบ custom สำหรับ Cloud Provider เฉพาะราย

- Real-time Dashboard ที่ใช้ WebSocket หรือเทคโนโลยีใกล้เคียง  
  > Dashboard ในเฟสนี้เน้นการแสดงผลสถานะ ณ เวลาปัจจุบันแบบไม่ต่อเนื่อง (non-streaming)
